# Best practices for MCP + Codex / Claude Code

## Цель

Этот документ описывает практический operating model для ускорения работы с MCP-памятью и агентами (Codex / Claude Code), чтобы:

- минимизировать потерю контекста между сессиями;
- снизить число повторных исследований;
- повысить качество архитектурных решений;
- удерживать предсказуемую стоимость по токенам/времени.

---

## 1) Базовый цикл каждой сессии

1. В начале сессии выставлять контекст проекта: `set_context(collection="levara")`.
2. Затем поднимать краткую память: `wake_up(max_tokens=300)`.
3. Перед новым исследованием сначала делать recall: `recall_memory(...)` / `list_memories(...)`.
4. Любые значимые решения/дискавери сохранять сразу, не в конце работы.

> Принцип: **recall before research**.

---

## 2) Строгая схема памяти: room × hall

Каждая запись в память должна иметь:

- `room` — о чём факт (подсистема/тема);
- `hall` — жанр факта.

Рекомендуемые `hall`:

- `fact` — стабильные факты (IP, endpoint, версии);
- `event` — событие во времени (обязательно абсолютная дата);
- `decision` — выбор + почему;
- `preference` — пользовательские предпочтения;
- `advice` — повторно применимое правило;
- `discovery` — симптом + корень проблемы + фикс.

### Guardrails

- Запрещать сохранение с пустыми `room/hall`.
- Не сохранять код, пути файлов и git-историю в память.
- Не сохранять относительные даты ("вчера", "прошлая неделя").

---

## 3) Когда обязательно вызывать `save_memory`

Вызывать `save_memory` немедленно, если:

- принято архитектурное решение;
- найден root cause после отладки;
- пользователь уточнил стиль/ограничения;
- появился новый сервис/endpoint/IP/порт;
- закрыт важный этап (feature/refactor/release);
- сформулировано переиспользуемое техническое правило.

---

## 4) Pin policy (чтобы wake_up не зашумлялся)

- `priority=10` — глобальные пользовательские правила и стиль;
- `priority=8` — критичная инфраструктура;
- `priority=5` — текущие крупные архитектурные решения;
- `priority=1..3` — второстепенные подсказки.

Раз в неделю делать ревизию pinned записей и убирать устаревшие (`unpin_memory`).

---

## 5) Поиск и retrieval в больших коллекциях

Для крупных баз по умолчанию использовать фильтры:

- `room` (обязателен при рабочем поиске);
- `tags` (если есть предметная классификация).

Для MCP tools с большими ответами вводить:

- `limit` / `cursor` / пагинацию;
- режимы `brief` и `full`;
- server-side summary перед возвратом больших payload.

---

## 6) Temporal knowledge graph как источник правды

Для сущностей, которые меняются во времени (owner, статус, расположение), использовать temporal-подход:

- `query_entity(name="X")` — текущее состояние;
- `query_entity(name="X", as_of="...")` — состояние на дату.

Это снижает конфликты в oncall/devops и ускоряет postmortem-анализ.

---

## 7) Разделение общей памяти и ролей

Если есть повторяющиеся роли (reviewer, architect, oncall), использовать отдельные diary namespace:

- `diary_write(agent="reviewer", ...)`
- `diary_read(agent="reviewer", ...)`

Так уменьшается шум в общей памяти проекта.

---

## 8) Рекомендации по Codex / Claude Code

### Для Claude Code

- Для проектных MCP серверов использовать project-shared конфигурацию.
- Для персональных экспериментальных серверов — user/local scope.
- Контролировать лимиты output/timeout, а не просто увеличивать их.

### Для Codex

- Для задач по OpenAI API использовать docs-first подход через официальный docs MCP.
- Держать единый AGENTS policy: recall first, затем исследование.

---

## 9) Метрики эффективности (обязательно)

Минимальный набор KPI:

1. **Retrieval quality**
   - recall@k по room/hall;
   - доля задач, где найден релевантный decision/discovery до начала нового ресерча.

2. **Memory hygiene**
   - доля записей с валидными room/hall (цель 100%);
   - доля pinned записей (не должна расти бесконтрольно).

3. **Agent efficiency**
   - time-to-first-correct-answer;
   - среднее число tool-calls на решённую задачу;
   - доля задач без повторного rework.

---

## 10) План внедрения 30/60/90

### 0–30 дней

- Включить обязательный bootstrap (`set_context` + `wake_up`) во все шаблоны.
- Внедрить client-side валидацию `room/hall`.
- Сделать recall-gate обязательным перед исследованием.

### 31–60 дней

- Добавить role diaries.
- Включить metadata-first search policy.
- Добавить `brief/full` режимы в тяжёлые MCP tools.

### 61–90 дней

- Собрать dashboard по retrieval/hygiene/efficiency.
- Ввести ежемесячный cleanup устаревших pinned записей.
- Формализовать шаблон архитектурных решений (decision + why + impact).

---

## 11) Короткий operational checklist

- [ ] Сессия начата через `set_context` + `wake_up`
- [ ] Перед новой работой выполнен recall
- [ ] Каждое решение/дискавери сохранено сразу
- [ ] Критичные факты pinned, остальное не pin
- [ ] Поиск идёт с `room/tags`
- [ ] В конце сессии сохранён краткий итог
