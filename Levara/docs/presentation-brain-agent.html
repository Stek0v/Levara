Проблема: инструменты есть, дисциплины нет

31 MCP-инструмент, но агенты не знают как и когда их использовать

Нет диагностики

Агент не может проверить здоровье системы программно. /health возвращает "ok" даже если embedding-сервер лежит.

Нет истории

Никакого лога событий. Невозможно понять: "когда последний раз прошёл sync?" или "сколько cognify упало за день?"

Нет протокола

Каждая сессия начинается с нуля. Агент не знает что читать перед ответом и что сохранять после.

Вдохновение: gbrain (github.com/garrytan/gbrain) — brain-first workflow для персональных агентов

NEW MCP Tool: heartbeat

Append-only лог системных событий для анализа трендов

Что логируется автоматически

  • doctor — результат каждой диагностики
  • cognify — завершение пайплайна (статус, chunks, entities, время)
  • sync — направление, remote URL, типы данных
  • prune — результат очистки графа

Хранение

  • Таблица heartbeats (PostgreSQL + SQLite)
  • Индекс по (event_type, created_at DESC)
  • REST: GET /api/v1/heartbeats?type=doctor&limit=10
// Запрос последних событий heartbeat( event_type: "cognify", limit: 5 ) // Ответ { "count": 3, "events": [ { "event_type": "cognify", "payload": { "collection": "docs", "status": "COMPLETED", "chunks": 42, "entities": 18, "elapsed_ms": 12340 }, "created_at": "2026-04-14T..." } ] }

Brain Playbook: протокол агента

Чёткие правила когда читать, когда писать, когда проверять

1

Ритуал старта

set_context → wake_up → doctor

Каждая сессия начинается с загрузки контекста и проверки здоровья

2

Читай-перед-ответом

recall_memory → search → query_entity

Проверь существующие знания до использования общих данных

3

Пиши-после-обучения

save_memory(room, hall)

Немедленно сохраняй факты, решения, инсайты с правильной классификацией

Правила-триггеры

ТриггерДействие
Принято архитектурное решениеsave_memory(hall="decision") с почему
Найден root cause багаsave_memory(hall="discovery")
Новый сервис/endpoint/портsave_memory(hall="fact") + pin
Завершена значимая работаsave_memory(hall="event") с датой

Кейсы использования

Реальные сценарии, где brain-agent layer даёт прирост

1. Onboarding нового разработчика

Загрузить docs + README через рецепт markdown → brain. Загрузить git-историю через git → brain. Агент отвечает на вопросы о проекте с цитатами из документации и коммитов.

2. Мониторинг деградации

Cron запускает doctor каждые 15 мин. Если embedding-сервис упал — heartbeat покажет тренд. Алерт до того, как пользователь заметит пустые ответы.

3. Кросс-проектный контекст

Каждый проект в своей коллекции с rooms. cross_search ищет по всем. Brain schema гарантирует единообразную структуру для фильтрации.

4. Audit trail решений

Все hall="decision" записи доступны через recall_memory(hall="decision"). Через полгода можно восстановить почему выбрали JWT вместо sessions.

5. Автоматическая синхронизация

Cron-профиль "продакшн": Mac и Pi синхронизируют memories и interactions каждые 15 минут. heartbeat логирует каждый sync.

6. Диагностика перед деплоем

doctor(verbose=true) перед деплоем новой версии. Проверяет что все зависимости на месте, BM25 индексы актуальны, граф не засорён orphans.

Лучшие практики

Memory hygiene

  • Всегда заполняй room + hall при save_memory
  • Пустые поля снижают полезность recall на ~40%
  • Пиней экономно: wake_up обрезается по бюджету токенов
  • Откреплять устаревшие записи через unpin_memory
  • НЕ сохранять пути к файлам — они stale через неделю

Doctor workflow

  • Запускай doctor в начале длинных сессий
  • Реагируй на remediation до начала работы
  • После крупного ingestion: doctor(verbose=true)
  • Мониторь тренды через heartbeat
  • В cron: алерт только на "status": "fail"

Рецепты

DO

Начинай с RAG-режима. Добавляй full mode только для важных документов.

DO

Устанавливай room/tags при ingestion. Фильтрованный поиск в 3x точнее.

DON'T

Не загружай >50KB за один вызов cognify. Разбивай на секции.

Плюсы и минусы

Плюсы

  • Агент может программно проверить здоровье всей системы за один вызов
  • Heartbeat даёт audit trail всех операций (cognify, sync, prune)
  • Playbook устраняет "холодный старт" — агент знает что делать
  • Brain schema обеспечивает единообразие между проектами и коллекциями
  • Cron-профили позволяют обнаружить деградацию до жалобы пользователя
  • Рецепты снижают порог входа для новых источников данных
  • Всё обратно совместимо — ядро хранения не затронуто
  • Doctor remediation на русском — actionable, не просто "check failed"

Минусы / ограничения

  • Doctor не в realtime — это snapshot на момент вызова
  • Heartbeat растёт бесконечно (нет auto-cleanup пока)
  • Cron внешний — нет встроенного scheduler в Levara
  • Нет push-уведомлений — только pull через heartbeat
  • Рецепты пока только 2 (git, markdown), нет email/calendar
  • Neo4j check через HTTP ping :7474, не полноценный driver test
  • Memory staleness считает по room, не по отдельным записям
  • Playbook — это документ, не enforce (агент может проигнорировать)
