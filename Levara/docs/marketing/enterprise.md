# Levara Enterprise — ИИ-память под корпоративный контроль

**Удобство ИИ-памяти — вместе с изоляцией, аудитом и управляемой идентичностью,
которые требует организация.**

## Для кого

Организации с требованиями к управлению доступом, изоляции подразделений, аудиту
и соответствию (compliance).

## Боль

- ИИ-инструменты удобны, но «куда уходят данные?» — стоп-фактор для службы
  безопасности.
- Нужна изоляция между командами/подразделениями (tenant) и доказуемый аудит.
- Идентичность должна управляться централизованно (корпоративный вход), а не
  россыпью паролей.

## Что даёт Enterprise (уже реализовано)

| Возможность | Что это даёт организации |
|---|---|
| Изоляция тенантов (tenant) с проверкой членства | Данные подразделений не пересекаются; доступ — только после проверки принадлежности |
| Параметризованные tenant-фильтры | Защита от подмены запроса (нельзя «склеить» доступ строкой) |
| Экспорт аудита (асинхронный, в JSONL) | Поток событий во внешние системы — основа для комплаенса |
| Адаптер OIDC (проверенные claims) | Подключение к корпоративной идентичности через мост идентичности |
| Строгий режим запуска (fail-fast) | Сервер не стартует без авторизации/SSO, изоляции тенантов и приёмника аудита |
| Контракты адаптеров хранилища и KMS | Готовые «разъёмы» для корпоративного хранилища и управления ключами |

## Для кого внутри компании

| Роль | Что ей важно | Сообщение |
|---|---|---|
| Platform Engineering | один self-hosted слой памяти для агентов | один Go-сервис, профили запуска, API/MCP/gRPC surfaces |
| Security | контроль доступа, tenant isolation, audit trail | strict mode, `pkg/access`, audit export, security checklist |
| Data/AI team | поиск, KG, rerank, workspace context | HNSW + BM25 + graph + Markdown truth layer |
| Compliance | понятная граница roadmap vs готово | готовые seams, но без ложного обещания KMS/SIEM production backends |

## Что в разработке (честно)

Чтобы вы планировали без иллюзий — следующее заложено как контракты и «разъёмы»,
но **конкретные реализации ещё в работе**:

- HTTP-поверхности SAML и SCIM (provisioning);
- конкретные бэкенды корпоративного объектного хранилища (S3 / GCS / Azure Blob);
- реализации KMS / BYOK (управление и «обёртывание» ключей);
- приёмник SIEM и принудительное исполнение legal hold.

Иными словами: **архитектурный фундамент и границы готовы**, конкретные
корпоративные бэкенды подключаются по мере дорожной карты. Не стоит считать пресет
Enterprise доказательством готовности KMS/BYOK или корпоративного хранилища к
продакшену.

## Сценарий

Несколько подразделений работают в одной инсталляции, но видят только свои данные.
Вход — через корпоративный OIDC. Каждое значимое действие уходит в поток аудита
для службы безопасности. Строгий режим гарантирует, что инсталляция без изоляции
тенантов или приёмника аудита просто не запустится.

## Доказательства в продукте

- `LEVARA_PROFILE=enterprise` в strict mode требует Postgres, auth или SSO bridge,
  stable signing config, tenant enforcement и audit sink.
- OIDC adapter принимает уже проверенные claims и мапит issuer/subject/groups
  через `pkg/access.IdentityBridge`.
- SCIM-shaped provisioner seam уже есть, но HTTP SCIM surface отложен до ADR.
- `pkg/storage` задаёт metadata/KMS contracts: tenant/project scope, digest,
  retention class, legal hold flag, encryption key reference.
- `docs/security-diff-checklist.md` задаёт блокирующие условия для access,
  tenant, audit, storage/KMS и MCP memory ownership изменений.

## Почему Levara

- **Governance с самого старта** — изоляция, аудит и строгая валидация уже в ядре.
- **Честная дорожная карта** — мы прямо разделяем «готово» и «в работе».
- **Тот же движок** — без форков и отдельных «корпоративных» сборок: enterprise —
  это слой адаптеров, а не другой продукт.
- **Производительность инженерного класса** — около 2,6 мс на поиск, сотни
  запросов в секунду, 100% восстановление после сбоя.

## Как начать (пилот)

```bash
cp deploy/profiles/enterprise.strict.env.example .env
# PostgreSQL, обязательная авторизация или SSO-мост,
# включённая изоляция тенантов, приёмник аудита
./levara-server -config-check
```

Для оценки безопасности изменений используйте `docs/security-diff-checklist.md`.
