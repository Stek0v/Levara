# Levara — Селфхостед MCP-сервер памяти для AI-ассистентов разработки

> Один бинарник на Go. Граф знаний + векторный + гибридный поиск. 15 MCP-инструментов.
> Работает на Raspberry Pi 5. Ноль облачных зависимостей. Ваш код остаётся вашим.

---

## Зачем это нужно

AI-ассистенты для разработки (Claude Code, Cursor, Windsurf) — мощные инструменты, но они забывают всё между сессиями. Каждый новый диалог начинается с нуля. Ассистент заново читает файлы, заново открывает архитектуру, заново узнаёт ваши соглашения.

Встроенные системы памяти примитивны:
- **Claude Code**: файлы CLAUDE.md (статический текст, ведётся вручную)
- **Cursor**: .cursorrules + @codebase (индексация на уровне файлов, без связей между сущностями)
- **Windsurf**: память workspace (нестабильная, без семантического поиска)

Ни один из них не предлагает:
- Семантический поиск по накопленным знаниям проекта
- Графы сущностей и связей (кто вызывает что, что от чего зависит)
- Сохранение решений и контекста между сессиями
- Осведомлённость об истории Git
- Изоляцию по проектам с возможностью командного доступа
- Приватность (все данные уходят на облачные серверы провайдеров)

**Levara решает все эти проблемы** как селфхостед MCP-сервер памяти.

## Архитектура

```
Машина разработчика                       Сервер Levara (Pi 5 / локальный / VPS)
┌──────────────────┐                     ┌─────────────────────────────────┐
│ Claude Code      │                     │ Levara (один Go-бинарник, 30МБ) │
│ или Cursor       │   MCP (HTTP)        │                                 │
│                  │ ◄─────────────────► │  ┌─ MCP-сервер (15 инструментов)│
│ .mcp.json        │                     │  ├─ HNSW векторный поиск        │
│ настроен на      │                     │  ├─ BM25 поиск по ключевым словам│
│ эндпоинт Levara  │                     │  ├─ Граф знаний (SQL)           │
└──────────────────┘                     │  ├─ Пайплайн Cognify            │
                                         │  ├─ Хранилище памяти            │
                                         │  ├─ Хранение чатов              │
                                         │  ├─ Анализатор Git              │
                                         │  └─ WAL-отказоустойчивость      │
                                         │                                 │
                                         │  Коллекции:                     │
                                         │   topka/     (проект A)         │
                                         │   levara/    (проект B)         │
                                         │   my-app/    (проект C)         │
                                         │                                 │
                                         │  Ollama (локальная LLM + embed):│
                                         │   qwen3:0.6b (522МБ)           │
                                         │   nomic-embed-text (274МБ)     │
                                         └─────────────────────────────────┘
```

**Одна папка проекта = одна коллекция.** Каждая коллекция — это изолированный HNSW-индекс + WAL + граф. Поиск в коллекции A никогда не возвращает данные из коллекции B.

## Какие проблемы решает

| Проблема | Без Levara | С Levara |
|----------|-----------|----------|
| **Переполнение контекстного окна** | Вся кодовая база не помещается; ручной выбор файлов | Семантический поиск извлекает только релевантные фрагменты по запросу |
| **Амнезия между сессиями** | Каждый разговор начинается с нуля | Накопленные знания сохраняются: решения, архитектура, паттерны |
| **Нет понимания архитектуры** | Приходится перечитывать код каждую сессию | Граф знаний с сущностями и связями сохраняется между сессиями |
| **Потерянные решения** | «Почему мы выбрали X?» — утеряно после сессии | Решения сохранены как память, доступны для запроса навсегда |
| **Нет осведомлённости о Git** | Ассистент не знает историю коммитов | Анализ коммитов: кто что менял, когда, зачем |
| **Нет командного доступа** | Память только у одного пользователя, на одной машине | Общий сервер, вся команда запрашивает одну базу знаний |
| **Проблемы приватности** | Облачные сервисы памяти видят ваш код | Селфхостед, данные никогда не покидают вашу сеть |
| **Привязка к вендору** | Зависимость от формата памяти конкретного ассистента | Стандартный протокол MCP — работает с любым MCP-клиентом |

## 15 MCP-инструментов

Levara предоставляет 15 инструментов через MCP (JSON-RPC 2.0 по HTTP):

### Загрузка знаний

| Инструмент | Что делает | Пример |
|------------|-----------|--------|
| **cognify** | Преобразует текст в граф знаний (сущности + триплеты + векторы) | `cognify(data="содержимое README.md", collection="my-app")` |
| **add** | Подготавливает текст для последующей обработки | `add(data="API спецификация", dataset_name="api-docs")` |
| **analyze_commits** | Парсит историю Git и превращает в поисковую базу знаний | `analyze_commits(repo_path="/home/dev/my-app", since="2024-01-01")` |

### Поиск

| Инструмент | Что делает | Пример |
|------------|-----------|--------|
| **search** | Мультистратегийный поиск (7 режимов) | `search(query="auth middleware", search_type="HYBRID", collection="my-app")` |
| **git_search** | Поиск по проанализированной истории коммитов | `git_search(query="кто менял схему БД")` |

### Память

| Инструмент | Что делает | Пример |
|------------|-----------|--------|
| **save_memory** | Сохраняет ключ-значение в память (user/project/feedback) | `save_memory(key="tech_stack", value="Next.js + Go", type="project")` |
| **recall_memory** | Семантический поиск по сохранённой памяти | `recall_memory(query="какой фреймворк")` |
| **list_memories** | Список всей сохранённой памяти | `list_memories(type="project", collection="my-app")` |

### История чатов

| Инструмент | Что делает | Пример |
|------------|-----------|--------|
| **save_chat** | Сохраняет сообщения диалога | `save_chat(session_id="abc", messages=[...])` |
| **recall_chat** | Загружает чат по ID сессии | `recall_chat(session_id="abc")` |
| **search_chats** | Полнотекстовый поиск по всем чатам | `search_chats(query="миграция базы данных")` |

### Управление

| Инструмент | Что делает |
|------------|-----------|
| **list_data** | Перечисляет датасеты и коллекции |
| **delete** | Удаляет датасет по ID |
| **prune** | Полный сброс системы |
| **cognify_status** | Проверяет прогресс асинхронного пайплайна |

## 15 типов поиска

Levara не ограничивается векторным поиском. Система предлагает 15 стратегий, комбинирующих различные сигналы:

| Тип | Как работает | Лучше всего подходит для |
|-----|-------------|--------------------------|
| **CHUNKS** | Чистый векторный поиск по сходству (HNSW) | «Найди код, похожий на X» |
| **CHUNKS_LEXICAL** | BM25 поиск по ключевым словам | Точные идентификаторы, сообщения об ошибках |
| **HYBRID** | Вектор + BM25 с RRF-слиянием | Универсальный (рекомендуется по умолчанию) |
| **RAG_COMPLETION** | Векторный поиск → LLM генерирует ответ | «Как работает аутентификация?» |
| **GRAPH_COMPLETION** | Вектор → извлечение сущностей → 1-hop по графу → LLM | «Что зависит от UserService?» |
| **GRAPH_COMPLETION_CONTEXT_EXTENSION** | Обход графа на 2 хопа → LLM | Глубокое исследование связей |
| **GRAPH_COMPLETION_COT** | Цепочка рассуждений: декомпозиция → мульти-поиск → синтез | Сложные вопросы, требующие рассуждений |
| **TRIPLET_COMPLETION** | Поиск по триплетам субъект-предикат-объект → LLM | «Что вызывает что?» |
| **NATURAL_LANGUAGE** | LLM преобразует вопрос в структурированный запрос → выполнение | Естественный язык по графу знаний |
| **CYPHER** | Прямой запрос к графу (синтаксис Cypher) | Продвинутые запросы к графу |
| **TEMPORAL** | Поиск с учётом времени и извлечением дат | «Что изменилось на прошлой неделе?» |
| **CODING_RULES** | Поиск по сущностям кода (функции, классы, импорты) | Соглашения кода, паттерны |
| **SUMMARIES** | Поиск по саммари документов | Высокоуровневый обзор |
| **FEELING_LUCKY** | Автоматический выбор лучшей стратегии | «Просто найди» |

## Пайплайн Cognify

При вызове `cognify` Levara преобразует сырой текст в структурированные знания:

```
Вход: "UserService аутентифицирует через JWT. Он вызывает TokenStore,
       который использует Redis для хранения сессий."
                    │
                    ▼
        ┌─── Чанкинг ───┐
        │ Разбивка по абзацам, функциям или предложениям │
        └────────┬───────┘
                 │
                 ▼
        ┌─── Извлечение через LLM ───┐
        │ qwen3:0.6b извлекает:                           │
        │ Сущности: UserService, JWT, TokenStore, Redis    │
        │ Связи: AUTHENTICATES_VIA, CALLS, USES            │
        └────────┬───────────────────┘
                 │
          ┌──────┴──────┐
          ▼             ▼
   ┌─ Эмбеддинг ─┐  ┌─ Запись в граф ─┐
   │ nomic-embed  │  │ graph_nodes     │
   │ → HNSW индекс│  │ graph_edges     │
   └──────────────┘  └────────────────┘
```

**Результат**: текст доступен для поиска по смыслу (вектор), по ключевым словам (BM25) и по связям (граф). Все три способа в одном запросе.

## Интеграция с Claude Code

### Конфигурация

`.mcp.json` в корне проекта (шарится через git):
```json
{
  "mcpServers": {
    "levara": {
      "type": "http",
      "url": "http://raspberrypi.local:8080/mcp"
    }
  }
}
```

Или через CLI:
```bash
claude mcp add --transport http levara http://raspberrypi.local:8080/mcp
```

### Как это работает на практике

1. **Разработчик открывает проект** → Claude Code подключается к Levara MCP
2. **«Как работает аутентификация?»** → Claude вызывает `search("auth", type="GRAPH_COMPLETION", collection="my-app")` → Levara возвращает сущности + связи + релевантный код → Claude отвечает с учётом знаний конкретного проекта
3. **Разработчик сохраняет решение** → Claude вызывает `save_memory(key="auth_choice", value="JWT с httpOnly cookies, обновление через Redis", type="project")`
4. **Через неделю, другая сессия** → Claude вызывает `recall_memory("auth")` → получает сохранённое решение, остаётся консистентным
5. **После спринта** → `analyze_commits(repo_path=".", since="2024-03-01")` → все коммиты проиндексированы, доступны для поиска по автору, файлу или типу изменений

### Типичный сценарий использования

```
Пользователь: "Объясни поток оплаты в этом проекте"

Claude (внутренне):
  1. Вызывает search("payment flow", type="GRAPH_COMPLETION", collection="payments-api")
  2. Получает: PaymentService → вызывает StripeGateway → генерирует событие PaymentCompleted
         → NotificationService слушает → отправляет чек по email
  3. Синтезирует ответ из контекста графа + фрагментов кода

Пользователь: "Запомни: мы мигрируем со Stripe на Adyen в следующем квартале"

Claude (внутренне):
  1. Вызывает save_memory(key="payment_migration", value="Миграция со Stripe на Adyen, Q2 2024", type="project", collection="payments-api")

Пользователь (через месяц): "Какой платёжный провайдер использовать для нового кода?"

Claude (внутренне):
  1. Вызывает recall_memory("платёжный провайдер", collection="payments-api")
  2. Получает: "Миграция со Stripe на Adyen, Q2 2024"
  3. Отвечает: "Используй Adyen — команда приняла решение мигрировать со Stripe."
```

### Почему Levara побеждает в категории self-hosted

1. **Ноль зависимостей**: один `go build` → один бинарник. Без Python, Docker, Redis, Neo4j, Qdrant. Развёртывание за 30 секунд.
2. **Развёртывание на периферии**: проверено на Raspberry Pi 5 (8ГБ, $80). Ни одно другое решение не запускает полный граф знаний + вектор + BM25 на одноплатнике за $80.
3. **Глубина поиска**: 15 типов поиска, включая цепочку рассуждений (COT), обход графа, расширение контекста. Конкуренты предлагают 1-5.
4. **Производительность**: 2.6мс поиск, 719 QPS. Вызовы MCP-инструментов незаметны на фоне 2-5-секундного инференса LLM.
5. **Приватность по умолчанию**: ваш код, ваше оборудование, ваша сеть. Ноль облачной телеметрии.

## Развёртывание на Raspberry Pi 5

### Оборудование
- Raspberry Pi 5, 8ГБ ОЗУ
- microSD или NVMe HAT (NVMe рекомендуется)
- Стоимость: ~$80 единоразово

### Бюджет ресурсов

| Компонент | ОЗУ |
|-----------|-----|
| ОС + среда выполнения | 1.5ГБ |
| Ollama + qwen3:0.6b (LLM) | 522МБ |
| Ollama + nomic-embed-text (эмбеддинги) | 274МБ |
| Levara (HNSW + WAL + граф) | 50-200МБ |
| **Свободно** | **~5.5ГБ** |

### Установка (5 минут)

```bash
# 1. Установить Ollama
curl -fsSL https://ollama.ai/install.sh | sh
ollama pull qwen3:0.6b
ollama pull nomic-embed-text

# 2. Скачать Levara
wget https://github.com/stek0v/levara/releases/latest/download/levara-linux-arm64
chmod +x levara-linux-arm64
mv levara-linux-arm64 /usr/local/bin/levara

# 3. Запустить
export OLLAMA_MAX_LOADED_MODELS=2  # критично: обе модели в памяти
export DB_PROVIDER=sqlite
export DB_PATH=$HOME/levara/data/levara.db
export EMBEDDING_ENDPOINT=http://localhost:11434/v1/embeddings
export EMBEDDING_MODEL=nomic-embed-text
export LLM_ENDPOINT=http://localhost:11434/v1
export LLM_MODEL=qwen3:0.6b

levara -standalone=true -dim=768 -shards=1 -port=8080 -data-dir=$HOME/levara/data
```

### Производительность на Pi 5

| Метрика | Значение |
|---------|----------|
| Задержка поиска (p50) | 2.87мс |
| Задержка поиска (p99) | 21.5мс |
| Конкурентный QPS | 652 |
| Время cognify (на документ) | ~25с |
| Вызов памяти | 3.45мс |
| Тест на реальных данных | 30/30 запросов, 100% попаданий |

## Мульти-нодовая репликация

Для команд или высокой доступности:

```bash
# Основная нода (принимает запись + обслуживает чтение)
levara -node-id=pi1 -port=8080

# Реплика (стримит WAL с основной, обслуживает чтение)
levara -node-id=pi2 -port=8080 -join-addr=pi1:8080
```

Реплики получают WAL-записи в реальном времени (~10-50мс задержки). Чтение может идти на любую ноду. Запись идёт на основную ноду (реплики перенаправляют автоматически).

## Когда использовать Levara

### Используйте Levara когда:
- Вам нужна постоянная, семантическая память между сессиями кодирования
- Вам важна приватность (код не уходит в облако)
- Вы работаете с несколькими проектами и нужна изоляция
- Вам нужна осведомлённость об истории Git в вашем ассистенте
- Вы работаете на минимальном оборудовании (Pi, небольшой VPS, изолированная сеть)
- Вам нужны связи в графе знаний, а не просто текстовый поиск

### Не используйте Levara когда:
- Вам нужен управляемый облачный сервис без администрирования (используйте mem0 Cloud)
- Вам нужна память, ориентированная на диалоги с профилированием пользователей (используйте Zep)
- У вас уже есть Neo4j и вы хотите зрелый Python-пайплайн Cognee
- Вам нужно только простое хранилище ключ-значение (используйте базовый MCP-сервер памяти)

| `OLLAMA_MAX_LOADED_MODELS` | 1 | Установить `2` (критично для Pi) |

## Технические детали

### Поток данных

```
Claude Code спрашивает: "Что изменилось в модуле авторизации?"
        │
        ▼
MCP-вызов: search("изменения модуля авторизации", type="HYBRID", collection="my-app")
        │
        ▼
Levara:
  1. Эмбеддинг запроса → nomic-embed-text → 768-мерный вектор  [0.3мс]
  2. HNSW поиск → top-K по косинусному сходству               [1.5мс]
  3. BM25 поиск → top-K по ключевой релевантности              [0.5мс]
  4. RRF-слияние → объединение и переранжирование               [0.1мс]
  5. Возврат top 5-10 результатов с метаданными                 [0.2мс]
        │                                                    ─────────
        ▼                                                    Итого: ~2.6мс
Claude получает контекст, генерирует ответ                     [2-5с LLM]
```

### Протокол MCP

Levara реализует MCP через JSON-RPC 2.0 по адресу `POST /mcp`:

```json
// Запрос
{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"search","arguments":{"search_query":"auth","collection":"my-app"}}}

// Ответ
{"jsonrpc":"2.0","id":1,"result":{"content":[{"type":"text","text":"[{\"id\":\"chunk-1\",\"score\":0.95,\"text\":\"UserService аутентифицирует через JWT...\"}]"}]}}
```

Поддерживаемые методы: `initialize`, `tools/list`, `tools/call`, `notifications/initialized`.

## Итоги

Levara — это селфхостед MCP-сервер памяти, который даёт AI-ассистентам для разработки постоянные, поисковые, привязанные к проектам знания. Он объединяет граф знаний, векторный поиск, BM25 и 15 стратегий поиска в одном Go-бинарнике, который работает на Raspberry Pi.

**Для разработчиков**: ваш ассистент наконец-то помнит то, что вы ему сказали на прошлой неделе.

**Для команд**: общая база знаний, к которой может обращаться ассистент каждого члена команды.

**Для предприятий**: ноль облачных зависимостей, полный контроль над данными, готовность к работе в изолированной сети.

```
pip install ничего
docker pull ничего
просто запустите бинарник
```
