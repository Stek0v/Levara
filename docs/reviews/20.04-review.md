## Executive Summary

Levara — зрелый production-проект с сильным ядром (HNSW + WAL + group-commit + Raft), чистой MCP-абстракцией (F-4 split завершён 2026-04-17) и адекватным WebUI на Next.js 15. Основные проблемы — на границах:

| Ось | Оценка | Главная болевая точка |
|---|---|---|
| **Ядро (store/arena/wal/hnsw/raft)** | 8.5 / 10 | Search держит RLock весь traversal — упирается throughput под write-mix |
| **Pipeline / orchestrator** | 7 / 10 | 1700+ строк в одном файле, silent panic в goroutine при cognify |
| **HTTP API** | 6 / 10 | `api.go` 1944 строк God-object, нет OpenAPI, нет rate-limit |
| **gRPC API** | 7 / 10 | Нет auth, proto без версии (levara.v1 — декларация, не механизм) |
| **MCP tools** | 8 / 10 | `BaseCognifyConfig()` течёт `orchestrator.Config` наружу |
| **WebUI** | 6.5 / 10 | 3 страницы с raw `fetch()`, нет auth-guard, settings не сохраняются |
| **Тесты** | 7 / 10 | `internal/http` handlers без unit-тестов, `pkg/router`/`rerank`/`graphrank` пустые |

**Top-3 блокеров перед релизом:**
1. **Auth guard во фронте** — незалогиненный юзер попадает в `/datasets`, ловит 401 на каждом запросе и не понимает почему. WebUI не зовёт `/auth/me` при загрузке и не редиректит на `/login`.
2. **Нет rate-limiting ни на одном канале** — HTTP/gRPC/MCP открыты для abuse.
3. **Proto без реального versioning** — `package levara.v1` без `reserved` полей и migration path. Любой breaking change ломает существующих gRPC клиентов.

**Top-3 архитектурных долга:**
1. Разбить `internal/http/api.go` (1944 строк) и `internal/http/graph_search.go` (1318 строк) на стратегии/хэндлеры.
2. Сузить `mcp.Deps.BaseCognifyConfig()` до narrow interface (сейчас течёт 30+ полей `orchestrator.Config`).
3. Embed-клиент кэшировать в `APIConfig` — сейчас создаётся заново на каждый поиск (`graph_search.go:30`), ест 10–50ms и TCP-коннекты.

## Архитектурный анализ

### 1. Слои и их границы

Архитектура строится на четырёх основных слоях:

**Хранилище** (`internal/store`): Ядро — `Levara` (17.7KB, ~650 строк) — моно-объект, объединяющий HNSW-индекс, WAL, arena памяти и метаданные на диске. `CollectionManager` управляет множественными `Levara`-экземплярами. Структура с `sync.RWMutex` + `pendingMu` обеспечивает двухфазную запись: данные идут в WAL/arena под `db.mu.Lock()`, HNSW-индексирование асинхронно в background (ограничивает время блокировки).

**Транспорт** (`internal/grpc`, `internal/http`): gRPC сервис (~220 строк) и HTTP handlers (~22.6KB) — должны быть чистыми маршрутизаторами без бизнес-логики. `Handler` (4.3KB) тонкий — делегирует `Cluster`. Однако Cognee API в `api.go` (**1944 строки**) — God-object: управляет datasets, cognify, поиском, синхронизацией, notebooks, ontologies. Это основной code smell проекта.

**Поиск** (`internal/http/graph_search.go`, `pkg/orchestrator`): **1318 строк** на граф-поиск — множественные стратегии (`GRAPH_COMPLETION`, `CYPHER`, `NATURAL_LANGUAGE`, `TRIPLET`). Композиция: embed → HNSW/BM25 (RRF) → graphrank → neo4j/postgres context → LLM. Много аллокаций в цикле, нет пулинга.

**MCP** (`pkg/mcp`, 40 Go-файлов, 19 тестов): Чистая contract-based зависимость через интерфейс `Deps`. Tools (search, cognify, memory, community, sync) разделены по файлам. F-4 wave рефакторинг завершён — минимальная surface. Но `Deps.BaseCognifyConfig()` возвращает `orchestrator.Config` (большой value), нарушая идею тонкого интерфейса.

**Кластеризация** (`internal/cluster`): Raft FSM + `DirectNode` для standalone. `ReplicationServer` для WAL streaming. Состояние: single source of truth в `db.Insert/Search`, Raft Apply вызывает FSM, который делает `db.Insert` на follower.

**Проблема границ**: Циклических зависимостей нет, но `internal/http/api.go` нарушает SRP. `graph_search.go` повторно вызывает pipeline (дублирование). Связанность высока через `APIConfig`.

### 2. Write path & concurrency

**Двухфазная запись** (`internal/store/db.go:199-260`):
1. Фаза 0: `disk.Write(metadata)` вне `db.mu` (~5μs выигрыш).
2. Фаза 1: `db.mu.Lock()` → `arena.Add` + `wal.WriteEntryNoFlush` + index update (~1–2μs hold).
3. Фаза 2: `wal.FlushAsync()` вне lock → group-commit `fsyncLoop` (2ms батч, ≤16 pending).

**Hold-time**: 27 `if err != nil` в db.go, все с ранним unlocking. Race-safe на практике благодаря:
- Batch hold time ~100μs (arena.Add + 2 map writes), fsync кооперативно вне lock.
- `fsyncLoop` (`wal.go:71-116`): таймер 2ms, батч ≤16 — группирует до 16 fsync в один.

Измерения из CLAUDE.md подтверждают линейное скалирование: 1→188 ips, 4→816 ips (4.3×), 8→1616 ips (8.6×).

**Lock-free чтение — не совсем**: `Search` (`db.go:394`) держит `db.mu.RLock()` весь путь через HNSW. `HNSW.Search` держит RWMutex весь traversal (fix F-6: раньше был race). На горячем пути — 200+ vector comparisons под hold. Для throughput-критичного можно оптимизировать: snapshot entry point, release lock, traverse с tombstone check.

**Скрытый артефакт** (`hnsw_race_test.go`): writes не конкурируют с search до `indexerLoop`-a — pending-очередь изолирует новые вставки от поиска. Для теста это ок, для реальной latency-distribution — значит, что search "видит" запись с задержкой indexer-loop.

### 3. Read path

**Композиция** (`graph_search.go:22-100`, api.go search handler):
1. **Embed**: query → 768D вектор (одна аллокация).
2. **Search**: HNSW (snap + RLock на 200 comparisons) + BM25 (trie-based, lock-free).
3. **RRF hybrid** — комбинирует HNSW + BM25 scores.
4. **GraphRank**: entity extraction → Neo4j queries (sync, 3s timeout — может зависнуть).
5. **Rerank** (optional): HTTP к внешней модели.
6. **Router adaptive weights** (`pkg/router`): feedback-driven learning, SELECT из DB **на каждый запрос** (N+1 issue).

**Горячий путь**:
- HNSW search: O(log N) traversals, RLock hold высокий.
- BM25: O(1) lookup + intersection, очень быстро.
- Аллокации: новый embed client создаётся **на каждый запрос** в `graph_search.go:30` — утечка TCP-соединений, невозможность переиспользовать пул.

**Branching**: много условных путей (Neo4j vs SQL fallback, LLM enabled/disabled, rerank yes/no) — CPU cache misses.

### 4. State management

**Источники правды:**
- **Primary**: `db.Insert` в `internal/store/db.go` — on-disk arena + WAL + BM25 index.
- **Replication**: WAL streaming (`cluster/replication.go`) + FSM Apply на followers.
- **PostgreSQL/SQLite**: метаданные в palace-таблицах (datasets, chunks, entities, relations) — **отстаёт** от векторного индекса (async cognify пишет).

**Синхронизация**:
- Insert/Delete → WAL + arena → pending → async `indexerLoop` → HNSW.
- Cognify (orchestrator) → entity extraction → Postgres write → самотриггеринг reindex.
- **WAL recovery** (`db.go:122-147`) — 2-pass: сначала собрать deletes, потом apply inserts (skip deleted). **Потенциально неправильно**: если одна WAL-запись Delete, а вторая Insert одного ID, recovery пропустит Insert. Нужна проверка порядка.

**Raft mode**: `FSM.Apply` вызывает `db.Insert`, логический порядок Raft сохраняется. Но `BatchInsert` может переупорядочиться внутри батча.

### 5. MCP package (pkg/mcp)

**Интерфейс Deps** (~40 строк): `DB()`, `Q()`, `Embed()`, `CollectionSearch()`, `RunPipeline()`, `NewSearchPipeline()` + 10 других. Clean seam.

**Нарушение инкапсуляции**: `BaseCognifyConfig()` возвращает полный `orchestrator.Config` (десятки полей) вместо narrow interface. Tool-авторам придётся читать orchestrator. **Реком:** вернуть `CognifyConfigBuilder` с только нужными методами.

**Покрытие tools**: 21 инструмент в `tools.go`, 19 `_test.go`. Но:
- `tool_community.go` (~3.6KB) не тестируется по-функционально.
- `tool_memory.go` (~9.6KB) сложный, нужна интеграция.
- `tool_cognify.go` (~7KB) полагается на `RunPipeline` mock, не тестирует реальный orchestrator.

**Утечка абстракций**: `internal/http/mcp.go` (**1227 строк**) — это не MCP, это HTTP-wrapper для MCP. Circular-risk: `mcp/deps.go` импортирует `pipeline`, которая импортирует embed, а MCP-tools зовут pipeline через `Deps.NewSearchPipeline()`.

### 6. Test coverage

**Сильное покрытие:**
- `internal/store`: 14 `_test.go` (arena, hnsw_race, db_concurrent, wal, collections, checkpoint).
- `internal/cluster`: replication_test, fsm_test, direct_test, chaos_test.
- `pkg/mcp`: 19 тестов (tool_search, tool_cognify, tool_memory, deps, session).
- **Итого**: 99 `_test.go` на 140 non-test файлов ≈ 41 % по файлам.

**Дыры:**
- `internal/http`: только 4 contract/helper теста, **нет тестов handlers** (Insert, Search, Cognify, Dataset).
- `pkg/orchestrator`: `pgupsert_test`, но основной `pipeline.go` (~37KB, 1700+ строк) функционально не покрыт.
- `pkg/router`, `pkg/rerank`, `pkg/graphrank` — `_test.go` отсутствуют.
- `internal/http/graph_search.go` (1318 строк) — только `query_entity_test`.

### 7. Code smells

**God objects:**
- `internal/http/api.go` (1944 строк): datasets, cognify, search, memify, sync — всё в одном файле.
- `internal/http/graph_search.go` (1318 строк): 6 search-стратегий без strategy-pattern.
- `internal/http/mcp.go` (1227 строк): JSON-RPC мультиплексор с 21 tool-хэндлером.

**Silent error handling:**
- `db.go:240-242`: если `wal.WriteEntryNoFlush` падает, `db.index[id]` уже выставлен → инвариант нарушен, но unlock + return error. **Race**: search может увидеть `id` в index, но не в WAL.
- `graph_search.go:74-77`: `graphdb.NewWriter` может вернуть error, но `if neoErr == nil` молча даёт nil context.
- `api.go` cognify: `go cognifyRunPipeline(...)` — паника в goroutine теряется (нет `recover`).

**Дублирование:**
- Embed-клиент создаётся на каждый поиск (`graph_search.go:30`, `dualsearch.go`).
- `Pipeline.SearchByText` + `SearchPipeline` живут параллельно в `graph_search.go` и `mcp/tool_search.go`.

**Неявные недоделки:** rerank integration (errors молчат), router feedback loop (`api.go:472` — только SELECT, без UPDATE-сценариев в тестах).

### 8. Top-5 архитектурных рекомендаций

**R1. Разделить `api.go` God-object** (1944 строк)
Путь: `internal/http/api_datasets.go`, `api_cognify.go`, `api_search.go`, `api_memory.go`, `api_sync.go`. Каждый файл <400 строк, легче тестировать. Разделение по `api.go:75-151` (registration calls).

**R2. Извлечь граф-search стратегии в interface**
```go
// internal/http/search_strategy.go
type SearchStrategy interface {
    Execute(ctx context.Context, cfg APIConfig, req CogneeSearchRequest) (fiber.Map, error)
}
```
Вместо 1318-строчного файла — 5 файлов по 250–370 строк.

**R3. Кэшировать Embed-client** (`graph_search.go:30`)
Добавить поле `APIConfig.EmbedClient *embed.Client`, инициализировать один раз в `main.go` (как LLM Provider). Экономия 10–50ms на запрос + переиспользование TCP-пула.

**R4. Сузить `BaseCognifyConfig()` interface** (`pkg/mcp/deps.go:116`)
Вместо `orchestrator.Config` value — `CognifyConfigBuilder` interface с `EmbedEndpoint()`, `EmbedModel()`, `LLMProvider()`, `Collections()`. Контракт явный, tool-авторам не надо читать orchestrator.

**R5. Добавить unit-тесты handlers** (`internal/http`)
Создать `handler_test.go` с mock Cluster + CollectionManager. Тестировать Insert/Search/Delete с timeout + error-injection. Scope ~200–300 строк.

## API анализ

Levara имеет **три параллельные API-поверхности** (HTTP, gRPC, MCP) с множественными дублирующимися функциями и **несогласованными контрактами**.

### 4. Consistency Issues

| Операция | HTTP | gRPC | MCP | Проблема |
|---|---|---|---|---|
| **Insert** | `POST /data` → `{inserted, failed, errors[]}` | `Insert()` + auto-BM25 | `add` (только в cognify-контексте) | Разные семантики: raw vs metadata-parsing vs ingest |
| **Search** | `POST /search/text` (router: auto mode) | `SearchByText`, `HybridSearch`, `GraphCompletionSearch` (3 отдельных) | `search` (10+ strategies merged) | HTTP: один endpoint + router, gRPC: отдельные RPCs |
| **Delete** | `POST /delete`, `BatchDelete` | `Delete()` + metadata cleanup | `delete` по dataset_id (не vector_id) | Разные granularity (vector vs dataset) |
| **Memory** | `/memories` | `LLMCacheGet/Put` (только LLM responses) | `save_memory` + hall/room | Разные storage и семантики |
| **Collection** | CRUD `/collections` | `CreateCollection`, `ListCollections` | Implicit через cognify default | HTTP/gRPC explicit, MCP implicit |
| **Error reporting** | `{status, errors[]}` body | `codes.InvalidArgument/NotFound` | JSON-RPC error `{code, message, data}` | **3 несовместимых формата** |

**Именования путаются:** Insert / Add / Save / Create означают разное в разных каналах (vector vs file vs memory vs collection).

### 5. Auth & Rate Limiting

- **JWT (HTTP):** HS256, 24h TTL, cookies + Authorization header.
- **API Keys (HTTP):** отдельное хранилище, `POST /auth/keys`.
- **gRPC:** нет встроенного auth — полагается на TLS/mTLS на deployment-уровне.
- **MCP:** session ID в `Mcp-Session-Id` header (stateful sessions в памяти).

**Rate limiting — отсутствует полностью**. Нет лимитеров ни на одном канале, только Prometheus metrics для наблюдения.

**RBAC:** есть (`dataset_shares` таблица, superuser flags, JWT с `user_id`), но без rate-per-user.

### 6. Observability

**Prometheus metrics (15+):**
- `levara_search_requests_total`, `levara_insert_requests_total`
- `levara_search_duration_seconds`, `levara_insert_duration_seconds` (histograms с P99)
- `levara_vectors_total`, `levara_raft_state` (gauges)
- `levara_search_requests_by_type_total` (vec: search_type, source)
- `levara_router_decision_seconds`
- `levara_mcp_tool_duration_seconds`, `levara_mcp_tool_requests_total` (per-tool)
- `levara_embed_requests_total`, `levara_llm_requests_total` (per-model)
- `levara_sync_operations_total`

**Coverage**: HTTP endpoints (Insert/Search/BatchInsert) и MCP tools инструментированы. **Пропуски**: Datasets, Memories, User endpoints без metrics.

**Tracing:** через `pkg/observe/langfuse.go` для LLM calls, но не full distributed trace.
**Logging:** `pkg/observe/observe.go` (Logger type) + ErrorTracker, но без structured JSON-логов для всех endpoints.

### 7. Contract Stability

**Proto breaking-changes risk:**
- `service LevaraService` без version suffix, `package levara.v1` — декларация, но нет v2 плана.
- Message fields используют field numbers (защита от reordering), но **нет `reserved`** для deprecated fields.
- Добавление required поля в существующие messages сломает старых клиентов.

**HTTP swagger/openapi:** отсутствует. Endpoints документированы комментариями (`// U1: Health`, `// U2: Datasets`), но без машиночитаемой спеки.

**MCP schema:** JSON Schema draft 2020-12 inline в `tools.go` `InputSchema`. **Нет OutputSchema.**

## Заключение

Проект находится в **хорошей форме для своей стадии**: ядро Levara (HNSW + WAL + Raft + group-commit) — сильное и well-tested, MCP-слой чистый после F-4 split, WebUI покрыт 61 E2E-тестом. Но границы между слоями затёрты:

- **HTTP-слой размыл свою роль** — должен быть тонким маршрутизатором, а стал 1944-строчным God object.
- **Три API-канала (HTTP/gRPC/MCP)** живут параллельно без единого контракта, с тремя разными форматами ошибок и двумя разными auth-моделями.
- **WebUI-to-backend интеграция** хороша в happy-path, но проваливается на edge-cases: 401, ошибки сети, settings persistence.

**Критический путь перед production:** auth-guard во фронте, rate-limiting на API, embed-client-кэш в APIConfig, разбиение `api.go` — эти 4 задачи закрывают большую часть operational-рисков и делают проект safe для внешних пользователей.
