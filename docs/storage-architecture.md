# Архитектура хранения Cognee + VectraDB

## Контекст

После деплоя Cognee + VectraDB + Qwen 3.5 на сервер 10.23.0.64 возникли вопросы:

1. **Можно ли менять модель?** Если переключить с локальной Qwen на облачную (GPT-4 и т.д.) — где хранится контекст? Будет ли он доступен?
2. **Дублирование хранения**: VectraDB и Neo4j хранят похожие данные — зачем оба? Можно ли обойтись одним?

---

## 1. Смена модели — что произойдёт с контекстом

### Где хранится «контекст» (память системы)

Контекст хранится **полностью локально** в 4 базах. Ни один бит не уходит в облако модели:

| Хранилище | Что хранит | Зависит от LLM? | Зависит от embedding модели? |
|-----------|-----------|:---:|:---:|
| **Neo4j** | Knowledge graph: ноды (`Person`, `Company`...) + рёбра (`WORKS_AT`, `USES`...) | Да* | Нет |
| **VectraDB** | Векторные эмбеддинги + метаданные (payload) | Нет | **Да** |
| **PostgreSQL** | Метаданные нод/рёбер, ACL, пайплайн-раны | Нет | Нет |
| **Filesystem** | Исходные файлы (PDF, TXT...) | Нет | Нет |

\* Neo4j зависит от LLM косвенно: **структура графа** (какие сущности и связи извлечены) определяется LLM при `cognify`. Но сам граф хранится локально.

### Можно ли менять LLM модель (Qwen ↔ GPT-4 ↔ Claude)?

**Да, свободно.** Меняется только `LLM_MODEL` + `LLM_ENDPOINT` в `.env`:

```bash
# Было (Qwen локально через Ollama)
LLM_MODEL=ollama/qwen3.5
LLM_PROVIDER=ollama
LLM_ENDPOINT=http://ollama:11434

# Стало (GPT-4 облачно)
LLM_MODEL=openai/gpt-4o-mini
LLM_PROVIDER=openai
LLM_ENDPOINT=
LLM_API_KEY=sk-...
```

Всё, что уже в базах — остаётся:

- Knowledge graph в Neo4j — **сохраняется** (model-independent по формату)
- Embeddings в VectraDB — **сохраняются** (зависят от embedding модели, не LLM)
- PostgreSQL метаданные — **сохраняются**

**Нюанс**: если после смены LLM запустить `cognify` на новых данных — новая модель может извлекать **другие** сущности/связи (GPT-4 может найти больше нюансов, чем Qwen). Но старый граф остаётся валидным.

### Можно ли менять embedding модель?

**Нет, без переиндексации нельзя.** Причина:

- Каждая embedding модель создаёт вектора **разной размерности** (nomic-embed-text = 768, OpenAI text-embedding-3-large = 3072)
- Вектора из разных моделей **несовместимы** — cosine similarity между ними бессмысленна
- VectraDB запускается с фиксированной размерностью: `-dim=${VECTRADB_DIM:-1536}`

**Решение** после смены embedding модели:

1. Обновить `VECTRADB_DIM` в `.env` под новую размерность
2. Очистить VectraDB (пересоздать volume)
3. Перезапустить `cognify` — он создаст новые эмбеддинги

Knowledge graph в Neo4j при этом **не затрагивается** — можно пропустить LLM-шаг и только переиндексировать вектора.

## 2. Дублирование VectraDB и Neo4j

### Что хранится в каждой базе

```
           VectraDB                          Neo4j
    ┌─────────────────────┐         ┌─────────────────────────┐
    │ Вектора (float[1536])│         │ Nodes:                  │
    │ + metadata:          │         │   Person(name, desc)    │
    │   {text, id, type}   │         │   Company(name, desc)   │
    │                      │         │   Topic(name, desc)     │
    │ Коллекции:           │         │                         │
    │  DocumentChunk_text  │         │ Edges:                  │
    │  Entity_name         │         │   WORKS_AT              │
    │  TextSummary_text    │         │   USES                  │
    │  Triplet_text        │         │   RELATED_TO            │
    └─────────────────────┘         └─────────────────────────┘

    Назначение:                      Назначение:
    "Найди похожее по смыслу"        "Пройди по связям"
```

### Есть ли дублирование?

**Частичное — но намеренное:**

| Данные | VectraDB | Neo4j | PostgreSQL | Зачем дублируется |
|--------|:---:|:---:|:---:|---|
| Имена сущностей | ✓ (metadata) | ✓ (свойство ноды) | ✓ (JSON в attributes) | Нужны для разных типов запросов |
| Связи (рёбра) | ✗ | ✓ | ✓ | Neo4j = обход, PG = аудит |
| Текст чанков | ✓ (вектор + payload) | ✗ | ✗ | Только в VectraDB |
| Саммари | ✓ (вектор + payload) | ✗ | ✗ | Только в VectraDB |
| Эмбеддинги | ✓ | ✗ | ✗ | Только в VectraDB |

### Можно ли убрать Neo4j и обойтись VectraDB?

**Нет. Они решают разные задачи:**

**VectraDB** — "семантическая близость": *"Найди всё, что похоже на 'HNSW indexing'"*
- Берёт запрос → делает embedding → ищет ближайшие вектора через HNSW-индекс
- Не знает, что `HNSW` **используется** в `VectraDB`, а `VectraDB` **написан на** `Go`

**Neo4j** — "обход графа": *"Что связано с VectraDB? А что связано с тем, что связано?"*
- `VectraDB --USES--> HNSW --IS_TYPE_OF--> ANN_Algorithm`
- `VectraDB --WRITTEN_IN--> Go --ALSO_USED_BY--> Kubernetes`
- Без графа невозможен `GRAPH_COMPLETION` — самый мощный тип поиска в Cognee

### Как это работает вместе (GRAPH_COMPLETION)

```
Запрос: "Какой алгоритм использует VectraDB?"

1. VectraDB (семантический поиск):
   → Находит триплет "VectraDB → USES → HNSW" по эмбеддингу запроса

2. Neo4j (обход графа):
   → Берёт найденные ноды
   → Расширяет контекст: "HNSW → IS_TYPE_OF → ANN, VectraDB → SUPPORTS → Sharding"
   → Возвращает подграф из 11 нод + 10 связей

3. LLM (генерация ответа):
   → Получает контекст из графа
   → Генерирует: "VectraDB uses HNSW indexing for approximate nearest neighbor search"
```

**Без Neo4j** шаг 2 невозможен — VectraDB вернёт только 1 триплет, без расширенного контекста.

### PostgreSQL — зачем третья база?

PostgreSQL хранит **метаданные для мультитенантности**:
- Какой пользователь создал ноду
- К какому датасету она принадлежит
- ACL (права доступа)
- История пайплайн-ранов

Это **инфраструктурный слой**, не дублирование данных.

## Сводная таблица

| Вопрос | Ответ |
|--------|-------|
| Можно менять LLM? | **Да**, свободно. Контекст хранится локально. |

## Диаграмма потока данных

```
                    ┌──────────────┐
                    │  Документы   │
                    │  (PDF, TXT)  │
                    └──────┬───────┘
                           │ cognee.add()
                           ▼
                    ┌──────────────┐
                    │   Cognee     │
                    │  Pipeline    │
                    └──┬───┬───┬──┘
                       │   │   │
          cognify      │   │   │
       ┌───────────────┘   │   └───────────────┐
       ▼                   ▼                   ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   Neo4j      │   │  VectraDB    │   │  PostgreSQL  │
│              │   │              │   │              │
│ Граф знаний  │   │  Эмбеддинги  │   │  Метаданные  │
│ (сущности,   │   │  (чанки,     │   │  (ACL, раны, │
│  связи)      │   │   саммари,   │   │   датасеты)  │
│              │   │   триплеты)  │   │              │
└──────┬───────┘   └──────┬───────┘   └──────────────┘
       │                  │
       └────────┬─────────┘
                │ cognee.search()
                ▼
        ┌──────────────┐
        │     LLM      │
        │  (ответ)     │
        └──────────────┘
```

## Конфигурация в docker-compose

Все 4 сервиса определены в `docker-compose.full-stack.yml`:

| Сервис | Образ | Порт | Volume |
|--------|-------|------|--------|
| `cognee` | Build from `./cognee` | 8000 | — |
| `vectradb` | Build from `./VectraDB` | 8080 | `vectradb-data` |
| `postgres` | `pgvector/pgvector:pg17` | 5432 | `postgres-data` |
| `neo4j` | `neo4j:5-community` | 7474, 7687 | `neo4j-data` |
| `redis` | `redis:7-alpine` | 6379 | `redis-data` |
| `prometheus` | `prom/prometheus` | 9090 | — |

Ключевые env-переменные для смены модели:

```bash
# LLM (меняется свободно)
LLM_MODEL=openai/gpt-4o-mini    # или ollama/qwen3.5, anthropic/claude-3.5-sonnet
LLM_PROVIDER=openai               # openai, ollama, anthropic
LLM_ENDPOINT=                     # URL для Ollama: http://ollama:11434

# Embedding (смена требует переиндексации)
EMBEDDING_MODEL=openai/text-embedding-3-large
EMBEDDING_PROVIDER=openai
VECTRADB_DIM=1536                 # ДОЛЖНА совпадать с выходом embedding модели
```
