# Levara Brain Playbook

Протокол поведения агента для последовательной и качественной работы со знаниями через MCP-инструменты Levara.

## Ритуал начала сессии

Каждый разговор, затрагивающий знания проекта:

1. `set_context(collection="<проект>")` — установить область для всех операций
2. `wake_up(max_tokens=300)` — загрузить запиненные факты + top entities графа
3. `doctor()` — (опционально, для длинных сессий) проверить здоровье системы перед глубокой работой

## Читай-перед-ответом

Перед ответом на любой доменный вопрос:

- `recall_memory(query="<тема>")` — проверить, знаем ли мы уже ответ
- `search(search_query="<вопрос>")` — искать в загруженных знаниях
- `query_entity(name="<сущность>")` — если вопрос о известной персоне/компании/концепции

Переходить к общим знаниям только если все три вернули пусто. Отметить пробел:
> "Существующих знаний по X не найдено — отвечаю на основе общих данных обучения."

## Пиши-после-обучения

Когда обнаруживаешь новый факт, решение или инсайт во время разговора:

**Немедленно** вызывай `save_memory` с:
- `key` — короткий идентификатор (например, "auth-jwt-decision")
- `value` — сам факт, включая **почему** для решений
- `room` — подтема (см. [brain-schema.md](brain-schema.md) для стандартных rooms)
- `hall` — жанр (контролируемый словарь):
  - `fact` — объективная характеристика (версия, IP, путь)
  - `event` — что-то произошло в конкретный момент (всегда с датой)
  - `decision` — архитектурный/проектный выбор с обоснованием
  - `preference` — предпочтение пользователя по стилю/инструментам/workflow
  - `advice` — переиспользуемая рекомендация
  - `discovery` — неочевидный инсайт, баг или подводный камень

Пинить критичные факты: `pin_memory(key="...", priority=8)` для инфраструктуры, `priority=10` для глобальных предпочтений.

## Entity-First обогащение

Перед ответом о конкретной персоне, компании или проекте:

1. `query_entity(name="Alice")` — получить текущее состояние из графа
2. Включить связи сущности в ответ
3. Если сущность не существует, но должна — отметить и сохранить после разговора

## Политика цитирования

При цитировании результатов поиска указывать источник:

```
Согласно загруженной документации [collection: docs, chunk_id: abc123], ...
```

Это помогает пользователям верифицировать утверждения и повышает доверие к базе знаний.

## Правила-триггеры

| Триггер | Действие |
|---------|----------|
| Пользователь принял архитектурное решение | `save_memory(hall="decision")` с **почему** |
| Найден root cause после расследования | `save_memory(hall="discovery")` |
| Пользователь поправил подход или дал стилевое указание | `save_memory(hall="preference")`, пинить если глобально |
| Появился новый сервис/endpoint/IP/порт | `save_memory(hall="fact")`, пинить если критично |
| Завершена значимая работа (фича, рефакторинг) | `save_memory(hall="event")` с датой |
| Дана переиспользуемая рекомендация | `save_memory(hall="advice")` |
| Пользователь упомянул дедлайн или зависимость | `save_memory(hall="event")` с абсолютной датой |

## Что НЕ сохранять

- Пути к файлам, имена функций, структуру кода — выводимо из `git`/`grep`
- Git-историю — `git log`/`git blame` авторитетны
- Эфемерные детали текущей задачи — используй tasks/plans
- Всё что уже есть в CLAUDE.md

## Частота проверок здоровья

- Начало длинных сессий: `doctor()` — реагировать на `warn`/`fail` remediation до начала глубокой работы
- После крупного ingestion: `doctor(verbose=true)` — проверить покрытие embedding/BM25
- Еженедельно: `check_drift()` — отловить смену модели эмбеддингов
- При деградации качества поиска: `heartbeat(event_type="doctor")` — проверить тренд здоровья

## Дневники агентов

Для специализированных subagents (reviewer, architect, monitor):

- `diary_write(agent="reviewer", key="...", value="...")` — изолированный namespace
- `diary_read(agent="reviewer", query="...")` — читает только свои записи
- Держит project memory чистой, позволяя агентам вести свой контекст
