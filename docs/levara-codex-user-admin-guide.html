1. Что такое Levara

Levara это долговременная проектная память для AI-агентов, MCP-клиентов и RAG-сценариев. Она помогает Codex, Claude, Cursor, Cline и другим инструментам не терять контекст, искать по коду и документам, помнить архитектурные решения и восстанавливать историю проекта между сессиями.

Если коротко, Levara превращает репозиторий, документы, заметки, решения команды и историю работы в структурированную память, доступную через REST и MCP.

Поиск

Vector search, BM25 и workspace search помогают находить смысловые и точные совпадения.

Память

Memory palace хранит факты, решения, открытия, советы и пользовательские предпочтения.

Связи

SQL graph и VSA помогают отвечать на вопросы вида “что с чем связано”.

2. Для кого эта документация

Solo-пользователь с Codex

Вы один работаете над проектом и хотите, чтобы Codex:

  • помнил архитектурные решения;
  • быстро находил нужные места в коде;
  • не просил заново объяснять проект;
  • искал по документам и заметкам;
  • понимал связи между подсистемами;
  • сохранял важные выводы после дебага.

Администратор команды

Вы настраиваете Levara для нескольких разработчиков и хотите:

  • подключить существующий код и документы;
  • дать разработчикам единый MCP memory layer;
  • контролировать доступы;
  • мониторить ingestion, indexing и search;
  • снизить зависимость от внешних сервисов;
  • дать AI-агентам надежную проектную память.

3. Что улучшается после включения SQL graph и VSA

Раньше граф знаний мог требовать отдельный Neo4j. Теперь основной путь может работать внутри SQL-базы Levara, а VSA добавляет быстрый индекс связей поверх graph facts.

Стабильность

  • меньше внешних зависимостей;
  • проще локальный запуск;
  • меньше точек отказа;
  • проще backup и sync;
  • меньше расхождений между памятью, графом и RAG.

Продуктивность

  • лучше ответы на вопросы “почему” и “как связано”;
  • быстрее восстановление контекста;
  • меньше повторного объяснения агенту;
  • лучше onboarding новых разработчиков;
  • контекстные code review и debug-сессии.
Практический смысл для пользователя: Levara становится не просто поиском по файлам, а долговременной памятью проекта, которая хранит решения, связи и актуальный контекст.

4. Базовые понятия

Dataset Логическая группа данных, прав доступа и graph facts. backend, frontend, project-alpha

VSA Быстрый внутренний индекс graph facts для поиска вероятных связей. Поиск кандидатов для subject + predicate -> object

5. Рекомендуемая архитектура

Solo setup

Codex Desktop / CLI | | MCP v Levara local server | v SQLite + vector collections + SQL graph + VSA

Подходит для одного разработчика, локального ноутбука и проектов, где важна простота.

Team setup

Developers / Codex / Cursor / Claude | | MCP / REST v Shared Levara Server | v Postgres or SQLite | v Collections + SQL graph + VSA + memories + audit

Подходит для командного сервера, нескольких разработчиков, централизованного backup и мониторинга.

6. Установка для solo-пользователя

Клонировать проект

git clone https://github.com/Stek0v/Levara.git
cd Levara

Собрать сервер

go build ./cmd/server

Запустить локально с SQLite

LEV_DB_PROVIDER=sqlite \
LEV_DB_PATH=./levara.db \
./server

Проверить health

curl http://localhost:8080/health
Для solo-сценария SQLite обычно достаточно: меньше настройки, меньше внешних сервисов, легче переносить и бэкапить.

7. Настройка MCP для Codex

Codex должен видеть Levara как MCP server. Точный способ зависит от окружения, но общий смысл один: MCP client должен уметь обращаться к Levara API.

Пример .mcp.json

{
  "mcpServers": {
    "levara": {
      "command": "levara-mcp",
      "args": [
        "--url",
        "http://localhost:8080/api/v1"
      ]
    }
  }
}

Проверить tools

После подключения должны быть доступны инструменты:

Группа Ключевые инструменты Когда использовать
Context set_context, wake_up, get_project_context, levara_instructions Старт сессии, загрузка pinned memories и правил агента.
Memory save_memory, recall_memory, list_memories, pin_memory, delete_memory, consolidate, consolidation_revert Долговременные факты, решения, открытия, preferences и уборка дублей.
Search/Graph search, cross_search, query_entity, list_communities RAG, graph-aware вопросы, поиск по нескольким collections.
Ingestion add, list_data, delete, prune, cognify, cognify_status, codify Загрузка данных, построение chunks, graph facts и code intelligence.
Workspace workspace_context, workspace_search, workspace_read, workspace_write, workspace_reconcile, workspace_conflicts, workspace_commit, workspace_revert Markdown-native project knowledge plane для Codex/Claude/команды.
Ops doctor, runtime_stats, ingestion_status, recent_errors, heartbeat, sync_status, workspace_ops_status Диагностика состояния instance, очередей, watcher, audit и sync.

Рекомендуемый старт каждой сессии

set_context(collection="levara")
wake_up(max_tokens=300)

Это загружает pinned memories и важный контекст проекта. Для работы с markdown workspace добавьте второй стартовый вызов: 

workspace_context(project_id="payments", branch="main")

Если агент отвечает на факты из workspace, правило строгое: workspace_search находит кандидатов, но перед ответом нужно выполнить workspace_read по citation.read_args.

8. Настройка для команды

Выбор базы

SQLite

Подходит для маленькой команды, proof of concept и single-machine deployment.

  • самый простой старт;
  • минимум зависимостей;
  • хорошо для локальной разработки.

Postgres

Подходит для нескольких разработчиков и централизованного сервера.

  • лучше concurrency;
  • привычный backup;
  • удобнее для production-like deployment.

LEV_STORAGE_PATH=/var/lib/levara/storage
LEV_WORKSPACE_PATH=/var/lib/levara/workspace

10. Markdown workspace knowledge plane

Новый workspace слой превращает markdown-файлы проекта в управляемую базу знаний для агентов. Это не просто upload документов: Levara хранит файлы по project_id и ветке, индексирует поколения, ведет manifest, проверяет freshness, возвращает citations, пишет audit log и дает агенту безопасный цикл “найти → точно прочитать → ответить”.

Базовый агентский цикл

workspace_context(project_id="payments", branch="main")
workspace_search(project_id="payments", branch="main", search_query="refund policy", search_type="HYBRID")
workspace_read(project_id="payments", branch="main", path="docs/refunds.md")
Правило источника истины: результат workspace_search приблизительный. Перед factual answer, архитектурным выводом или цитированием агент обязан прочитать точный markdown через workspace_read, обычно напрямую из citation.read_args.

Что возвращает workspace_context

Поле Зачем нужно
projectsТолько доступные пользователю workspace projects; ACL не раскрывает чужие проекты.
active_generationПоколение индекса, из которого по умолчанию отвечает поиск.
active_collectionВнутренняя collection, которую агенту не нужно угадывать вручную.
recommended_search_typeОбычно HYBRID: dense + BM25 для markdown corpus.
exact_read_requiredМашиночитаемый флаг правила “search then read”.
watcher и jobs_by_statusПоказывают pending watcher branches, ошибки и indexing backlog.
initialization_pathПодсказка для пустого проекта: write markdown, reconcile, затем search/read.

Запись и публикация markdown

workspace_write пишет файл в workspace filesystem truth и может сразу индексировать его. workspace_reconcile сверяет текущие markdown-файлы с индексом и публикует новое active generation. 

workspace_write(
  project_id="payments",
  branch="main",
  path="docs/refunds.md",
  text="# Refunds\n\nRefunds are handled by...",
  generation="gen-2026-06-22",
  activate_generation=true
)

workspace_reconcile(
  project_id="payments",
  branch="main",
  generation="gen-2026-06-22",
  activate_generation=true
)

CLI workflow

levara workspace context --project=payments --branch=main
printf '# Refunds\n\n...' | levara workspace write docs/refunds.md --project=payments --generation=gen-2026-06-22 --activate
levara workspace reconcile --project=payments --generation=gen-2026-06-22
levara workspace conflicts --project=payments
levara workspace manifest --project=payments

Freshness contract

workspace_search сам резолвит active generation и collection, если они не указаны. Ответ содержит freshness: можно понять, stale ли запрошенное поколение, есть ли pending watcher changes, когда был последний successful reconcile и сколько активных chunks/path.

Ситуация Что увидит агент Правильное действие
Запрошено старое поколение freshness.stale=true, reason requested_generation_is_not_active Не использовать как актуальный ответ без явной оговорки.
Watcher видит изменения, индекс еще не обновлен potentially_stale=true, watcher_branch_pending=true Выполнить workspace_reconcile или дождаться worker.
Нет active generation Ошибка с подсказкой run workspace_reconcile Создать/проиндексировать markdown и активировать generation.
В active generation несколько collections Ошибка неоднозначности Передать collection явно.

Citations и answer contract

Каждый hit обогащается workspace metadata: path, heading_path, file_digest, chunk_id, vector_id, generation, collection и citation.read_args. Это делает ответы проверяемыми: можно показать, из какого файла и digest сделан вывод. 

workspace_search(...)
→ result.citation.read_args

workspace_read(
  project_id="payments",
  branch="main",
  path="docs/refunds.md"
)

Durable indexing jobs

Для production Levara сохраняет indexing/reconcile jobs в durable outbox. Если процесс упал во время embedding или vector upsert, job останется видимым, будет retryable и при исчерпании попыток попадет в dead_letter. 

workspace_enqueue_index_job(
  project_id="payments",
  branch="main",
  operation="reconcile",
  generation="gen-2026-06-22",
  paths=["docs/refunds.md"]
)

workspace_index_jobs(project_id="payments", branch="main")
workspace_retry_index_job(project_id="payments", branch="main", job_id="...")

Полезные env-флаги:

LEVARA_WORKSPACE_INDEX_WORKER=1
LEVARA_WORKSPACE_WATCH_ASYNC_INDEX=1

Watcher, conflicts, commit и revert

Workspace хранит filesystem truth отдельно от active generation. Поэтому появились инструменты для drift/conflict detection и легкой истории изменений.

Инструмент Назначение
workspace_watch_statusСостояние watcher: pending branches, last scan/change/reconcile/error.
workspace_conflictsПоказывает dirty, unindexed и missing indexed markdown paths.
workspace_commitФиксирует snapshot измененных workspace files с message/author.
workspace_logПоказывает историю workspace commits.
workspace_revertВозвращает файлы к выбранному commit; после revert обычно нужен reconcile или reindex=true.
workspace_gcУдаляет stale generations/vector IDs/BM25 entries; dry-run показывает impact.

Context artifacts registry

Файл .kb/context-artifacts.json описывает важные артефакты вне обычных notes: OpenAPI, DDL, Terraform, ADR, runbook и другие markdown/source-файлы. Их можно перечислять, фильтровать по kind и переиндексировать отдельным workflow. 

{
  "version": 1,
  "includes": [
    {
      "project_id": "payments",
      "branch": "main",
      "glob": "docs/**/*.md",
      "kind": "markdown",
      "room": "docs",
      "tags": ["workspace"]
    },
    {
      "project_id": "payments",
      "glob": "api/**/*.yaml",
      "kind": "openapi",
      "room": "api",
      "tags": ["contract"]
    }
  ]
}
workspace_context_artifacts(project_id="payments", branch="main", kind="openapi")
workspace_reindex_artifacts(
  project_id="payments",
  branch="main",
  generation="gen-2026-06-22",
  kinds=["openapi"],
  activate_generation=true
)

11. Structured PDF extraction

Levara теперь умеет направлять table-heavy или visual-only PDF в schema-driven sidecar перед обычным ingest. Это полезно для счетов, актов, форм, сканов и PDF, где обычный текстовый extractor теряет таблицы или структуру.

Как работает pipeline

  1. Пользователь загружает PDF через /add или MCP add и передает schema, schema_id или structured_schema.
  2. Go server делает дешевый PDF preflight: ищет table signals, visual-only признаки и учитывает наличие schema.
  3. Если structured extraction уместен, Levara вызывает sidecar endpoint.
  4. Sidecar возвращает JSON extraction; Levara строит markdown projection и индексирует его как *.structured.md.
  5. Raw JSON сохраняется в structured_extractions/<data_id>.json рядом со storage.

Запуск sidecar

pip install lift-pdf
python3 scripts/lift_sidecar.py --host 127.0.0.1 --port 8097 --method vllm

Для локального HuggingFace inference:

pip install 'lift-pdf[hf]'
python3 scripts/lift_sidecar.py --method hf

Настройка Levara

export STRUCTURED_EXTRACT_ENDPOINT=http://127.0.0.1:8097/extract
# или
export LIFT_ENDPOINT=http://127.0.0.1:8097/extract

./server \
  --structured-extract-endpoint http://127.0.0.1:8097/extract \
  --structured-extract-timeout-ms 120000

Дополнительные env для sidecar:

VLLM_API_BASE=http://localhost:8000/v1
LIFT_METHOD=vllm
LIFT_MAX_OUTPUT_TOKENS=12384
LIFT_SIDECAR_PORT=8097

Upload пример

curl -X POST http://localhost:8080/api/v1/add \
  -F "dataset_id=finance-docs" \
  -F "schema_id=invoice-v1" \
  -F "files=@invoice.pdf"

В ответе появятся document_analysis и structured_extractions. Для plain text body structured extraction не запускается, но response все равно показывает document_analysis, чтобы клиент понимал выбранный pipeline.

Контракт sidecar

{
  "filename": "invoice.pdf",
  "content_base64": "...",
  "schema": "{\"type\":\"object\",\"properties\":{\"total\":{\"type\":\"number\"}}}",
  "page_range": [1]
}
{
  "extraction": {"total": 155.5},
  "metadata": {},
  "raw": "{\"total\":155.5}",
  "error": false,
  "token_count": 33
}
Structured extraction является optional capability. Если endpoint не настроен или preflight решает, что документ лучше обрабатывать обычным текстовым ingest, Levara продолжит стандартный pipeline.

12. Ежедневная работа

В начале дня

Восстанови контекст проекта через Levara и покажи последние важные решения.

Агент обычно выполнит:

set_context(collection="project-docs")
wake_up(max_tokens=500)
workspace_context(project_id="payments", branch="main")
recall_memory(query="recent architecture decisions", hall="decision")

Перед изменением кода

Что нужно знать перед изменением sync?

Levara может вернуть связанные memories, документы, graph entities и workspace results.

Перед ответом по документам

workspace_search(project_id="payments", branch="main", search_query="current refund SLA")
# затем взять citation.read_args из лучшего hit
workspace_read(project_id="payments", branch="main", path="docs/refunds.md")

Если freshness.potentially_stale=true, сначала выполните reconcile или явно предупредите, что ответ основан на не подтвержденном active generation.

Перед командной записью

workspace_conflicts(project_id="payments", branch="main")
workspace_write(..., expected_file_digest="sha256-from-last-read")
workspace_reconcile(project_id="payments", branch="main", generation="gen-2026-06-22", activate_generation=true)

После дебага

save_memory(
  room="workspace",
  hall="discovery",
  value="2026-06-17 workspace indexing failed on stale artifact manifests; fix is to reindex artifacts before retrying failed jobs."
)

После архитектурного решения

save_memory(
  room="graph",
  hall="decision",
  value="2026-06-17 chose SQL graph over mandatory Neo4j because deployment must work locally and on small servers with fewer dependencies."
)

13. Как работает RAG в Levara

Сигнал Для чего нужен Пример вопроса
Vector search Семантически похожие chunks. “Где обсуждалась эта идея?”
BM25 Точные совпадения и ключевые слова. “Где используется LEV_DB_PROVIDER?”
SQL graph Связи между сущностями. “Что связано с SQLite?”
VSA Быстрый поиск вероятных graph facts. “Какие отношения есть у этой сущности?”
Memory palace Решения, открытия, советы и preferences. “Что мы решили раньше?”
Пользователь не обязан понимать VSA. Достаточно знать, что Levara лучше ищет связи и быстрее восстанавливает контекст проекта.

14. Memory practices

Всегда указывать room и hall

Плохо:

save_memory(value="SQLite лучше")

Хорошо:

save_memory(
  room="storage",
  hall="decision",
  value="2026-06-17 chose SQLite for local-first deployments because it removes external DB dependency for solo users."
)

Hall vocabulary

Hall Когда использовать
factСтабильное объективное свойство.
decisionАрхитектурное или проектное решение и причина.
discoveryRoot cause, bug, gotcha или важное наблюдение.
adviceПовторно применимое практическое правило.
preferenceПользовательское предпочтение.
eventСобытие с абсолютной датой.
Не сохраняйте в memory секреты, большие куски кода, git history и пути файлов. Для этого есть repository, workspace search и git.

Pin policy

Priority Что пинить
10Глобальные пользовательские preferences и жесткие правила.
8Критичная инфраструктура: endpoints, IP, ports, protocol versions.
5Активные major decisions, которые нужны почти в каждой сессии.

Memory consolidation

Когда много похожих memories начинает засорять recall, используйте consolidate. Он планирует merge/supersede действий, проверяет summary на потерю чисел/сущностей и пишет consolidation_run_id, чтобы результат можно было откатить. 

consolidate(collection="levara", room="deploy", hall="discovery", dry_run=true)
consolidate(collection="levara", room="deploy", hall="discovery", dry_run=false)
consolidation_revert(run_id="...")

Recall before research

Перед архитектурным советом или изменением незнакомой подсистемы агент должен сначала искать прошлые решения и discoveries: 

recall_memory(query="workspace indexing", room="workspace", hall="decision")
recall_memory(query="migration failure", room="storage", hall="discovery")

17. Backup и sync

SQLite backup

sqlite3 levara.db ".backup levara-backup-$(date +%F).db"

Postgres backup

pg_dump "$POSTGRES_DSN" > levara-$(date +%F).sql

Что важно сохранять

  • graph_nodes и graph_edges;
  • memories;
  • data и dataset_data;
  • raw documents и storage;
  • workspace: markdown files, manifests, jobs, commits, audit jsonl и context-artifacts registry;
  • structured_extractions, если PDF extraction является источником audit/traceability;
  • vector collections, если их нельзя быстро пересоздать.

VSA можно перестроить из SQL graph, поэтому он важен для скорости, но не является самым критичным источником истины.

Sync

sync(
  remote_url="http://server:8080/api/v1",
  direction="pull"
)

По умолчанию sync ориентирован на memories, interactions и graph. Vector collections обычно не синхронизируются без явного opt-in, потому что требуют совместимых embedding dimensions/model и могут быть дешевле пересозданы через reindex/reconcile.

18. Безопасность

Рекомендации

  • Включить auth для командного сервера.
  • Использовать dataset-level access.
  • Разделять customer/project datasets.
  • Не давать всем write-доступ к общим memories.
  • Логировать MCP tool calls.
  • Для workspace использовать workspace_access_check перед сложными workflow.
  • Просматривать workspace_audit_log и workspace_ops_status при расследованиях.
  • Не хранить в audit snippets, search query, полный document text и exact file path там, где это запрещено политикой.
  • Не сохранять секреты в memory.
  • Не индексировать private keys, tokens и .env.

Audit privacy

Workspace audit events должны помогать расследовать “кто что сделал”, но не превращаться во вторую копию документов. События фиксируют user/bot id, project id, branch, operation, result и timestamp; они не должны хранить markdown text, snippets, search query или секреты.

19. Практические сценарии пользы

Новый разработчик

Просит Levara объяснить архитектуру проекта, ключевые решения и связанные документы.

Бонус: onboarding быстрее, меньше нагрузки на старших разработчиков.

Возврат через месяц

Codex восстанавливает контекст: что делали по VSA, SQL graph и отказу от Neo4j.

Бонус: меньше времени на восстановление истории.

Архитектурное изменение

Levara показывает связанные решения, риски, файлы и graph dependencies.

Бонус: решения принимаются на базе фактов.

Production incident

Агент ищет previous discoveries, runbooks, recent errors и sync status.

Бонус: быстрее root cause analysis.

Документированный ответ с citations

Агент ищет через workspace_search, читает exact markdown и отвечает с проверяемым source path/digest.

Бонус: меньше “галлюцинаций” и проще review решений.

Счета и табличные PDF

Structured extraction sidecar извлекает JSON по schema и индексирует markdown projection.

Бонус: поиск работает по нормализованным полям, а raw JSON остается в артефактах.

Workspace production checklist

Agent readiness

  • workspace_context возвращает active generation.
  • workspace_search возвращает freshness.
  • Search hits содержат citation.read_args.
  • workspace_read возвращает citation/chunk citations.
  • Agent instructions требуют exact read перед ответом.

Ops readiness

  • workspace_ops_status показывает zero-initialized jobs/audit.
  • workspace_index_jobs видит failed/dead_letter jobs.
  • workspace_watch_status показывает pending/error.
  • workspace_gc --dry-run понятен оператору.
  • Prometheus/Grafana смотрят job lag, dead letters и watcher pending.

Structured extraction checklist

  • Sidecar scripts/lift_sidecar.py запущен и доступен.
  • STRUCTURED_EXTRACT_ENDPOINT или LIFT_ENDPOINT настроен.
  • Timeout подходит для выбранного vLLM/HF backend.
  • Upload с schema_id возвращает document_analysis.
  • Для table-heavy PDF есть structured_extractions.status="ok".
  • Raw JSON artifact сохраняется в structured_extractions/<data_id>.json.
Главное сообщение для пользователей: Levara делает AI-агента полезнее не потому, что добавляет еще один поиск, а потому что дает ему долговременную проектную память: решения, причины, документы, связи, актуальные факты и прошлые открытия.
